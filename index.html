O problema · de onde viemos

Fora do padrão.

Três falhas no mesmo trecho: valor cru em vez de token, <nav> reimplementando um componente que já existe, e markup sem o design system.

gerado.tsx — fora do padrão
<div style={{ padding: '16px', color: '#1A2236' }}> <nav className="crumbs"> {trail.map(t => <a href={t.url}>{t.label}</a>)} </nav> </div> // #1A2236 já é color.text.default · 16px já é space.4 // <Breadcrumb> já existe em @mp/ui
no padrão
<Stack p="space.4" color="text.default"> <Breadcrumb items={trail} /> </Stack>
Por que passava pelo radar

O código roda.

Todos os checks de correção passam. Nenhum perguntava "isso usa o que já temos?". Correção não é conformidade — e só conformidade pega esses três casos.

ci · pull request

O problema de raiz · a guidance apodrecia

A skill define a regra.
O script, o inventário.

A skill de design system embutia tokens e props. A cada PR desatualizava e passava a informar o agente errado.

Agora a skill carrega só o porquê e o como. As minúcias saem do harvester.mjs. A skill referencia o output, não o copia.

skill rígida — apodrece
# tokens e props embutidos color.text.default: #1A2236 ← mentira quando muda componentes: Breadcrumb(items, separator) ← obsoleta a cada PR
skill = regra + referência
PORQUÊ consistência visual e acessível COMO sempre token · nunca valor cru · reuse @mp/ui INVENTÁRIO → .cache/{tokens,inventory}.json gerado por harvest.mjs · sempre atual

A classificação que evita falso-positivo

Cada checagem entra numa de três classes

GATE
bloqueia · determinístico
tsc · i18n/no-literal-string · color-no-hex · no-hardcoded-colors · no-duplicate-component · naming vs registry
AGENTE
julga · LLM (a frota #2661)
n+1 · waterfall · re-render evitável · injection · PII · conformidade com spec e skills
SINAL
avisa, nunca bloqueia
complexidade ciclomática · re-render rate (react-scan) · churn

Regra: o que não for determinístico não vira gate. Gate com falso-positivo ensina o time a dar bypass — e o harness perde valor.

Item 4 · PoC do scaffold · SpecUI

O scaffold, concreto:
spec → SpecUI → tela

SpecUI é spec-first e framework-agnostic: lê um spec YAML/JSON/TS e emite código via Generators + Templates — determinístico, encaixa no nosso boundary.

O pulo do gato: o generator default emite tags cruas (tag: button) — o que combatemos. O PoC usa um generator customizado ligado ao inventory.json + tokens.json: o scaffold já nasce no padrão.

specui.org · open source

assets-catalog.spec.yaml — entrada
screen: AssetsCatalog route: /assets-catalog tokens: { gap: space.4 } elements: - use: Breadcrumb # resolve via inventory.json items: $trail - use: DataTable columns: $cols
gerado · custom generator → @mp/ui
// SpecUI · determinístico · só componentes do catálogo import { Stack, Breadcrumb, DataTable } from '@mp/ui' export function AssetsCatalog() { return ( <Stack p="space.4"> <Breadcrumb items={trail} /> <DataTable columns={cols} /> </Stack> ) }
O que separa do Nível 4

Scaffold + template de spec + gates no CI.

A boa notícia: com catálogo, frontend.md e enforcement no lugar, o scaffold deixou de ser moonshot — virou trabalho de M. E o portal é read-only (mutationInvalidates vazio): o caso mais automatizável que temos.

Os mecanismos já existem. Falta ativá-los no gate e fechar a geração.
